# 2026-03-07 变更记录

## 概述

本次记录覆盖两轮连续迭代，重点解决了两个产品问题：

1. 登录链路不够顺，用户在跨设备验证、再次访问时仍有重复输入邮箱的摩擦。
2. 订阅页和推送机制还停留在“能用”，但没有达到稳定、轻量、低打扰的产品状态。

这两轮改动分别落在：

1. 长久登录 + 跨设备验证后的低摩擦登录体验
2. 推送分级/摘要机制 + 订阅页首屏重构

## 本次两轮改动

### 第一轮：登录体验整理

目标：

1. 让用户验证一次后尽量不再重复输入邮箱。
2. 支持手机和电脑任意设备点击邮件完成验证。
3. 验证成功后，另一台仍停留在订阅页的设备可以直接“一键登录”。
4. 弱化退出入口，强化“当前设备会保持登录”的心智。

实际完成：

1. 增加持久登录逻辑。
   - 浏览器本地保存 `portal_persistent_token` 和 `portal_persistent_email`
   - 再次访问订阅页时自动尝试恢复到门户
2. `/portal?token=...` 增加服务端校验。
   - 只有 `verified + active` 的订阅 token 才允许恢复登录
   - 无效 token 会回退到 `/subscribe?force=1`
3. 完成跨设备桥接登录。
   - 订阅/发送登录链接时生成短期 `bridge_ticket`
   - 任意设备点开邮件完成验证后，服务端会把 bridge 状态标记为已验证
   - 另一台设备轮询 bridge 状态，检测到已验证后出现“一键登录”按钮
4. 门户页调整账号操作层级。
   - 顶部 `切换账号` 改为更轻的 `更换邮箱`
   - 设置页底部加入“当前设备会保持登录”的说明
   - `退出当前设备` 改成低优先级文字按钮 `仅在本设备退出`
5. 登录提示更新。
   - 登录成功 toast 改成“已成功登录，下次打开会自动进入门户”
   - 订阅页新增跨设备验证完成后的直达登录提示

### 第二轮：推送机制与订阅页重构

目标：

1. 降低“每门课一封邮件”的打扰感。
2. 保留真正紧急课程的即时触达能力。
3. 让订阅页更像产品首页，而不是单纯的表单弹窗。

实际完成：

1. 推送机制引入 `delivery_mode` 概念。
   - `priority`：立即处理的课程，直接发
   - `digest_urgent`：近期课程摘要
   - `digest_soon`：普通新课摘要
   - `digest_daily`：每日汇总
2. 调度器聚合窗口缩短并参数化。
   - `PUSH_URGENT_DIGEST_MINUTES=5`
   - `PUSH_SOON_DIGEST_MINUTES=30`
3. 邮件标题和内容按推送场景变化。
   - 即时提醒、近期摘要、今日汇总、退课补录不再共用同一个标题
4. 邮件正文增加“你为什么收到这封邮件”的解释块。
   - 展示校区、类别、自主签到等偏好
5. 订阅页重构为双栏首屏。
   - 左侧：产品说明、核心价值、登录与推送机制说明
   - 右侧：原订阅表单与数据洞察卡片
6. 保持移动端兼容。
   - 小屏幕下自动回落为单列布局

## 过程日志

### 日志 1：登录体验改造

问题背景：

1. 用户验证邮箱后仍需要重新输入邮箱才能进入门户。
2. 用户可能在手机邮箱里点击验证，但电脑页面无法感知状态变化。
3. 退出入口过于显眼，容易破坏“长久登录”的预期。

设计决策：

1. 不做传统账号密码系统。
2. 继续沿用邮箱 magic link + token 的轻量方案。
3. 通过 `bridge_ticket` 做跨设备状态同步。
4. 通过本地持久 token 做“再次访问自动恢复登录”。

实现步骤：

1. 增加桥接票据模型 `LoginBridgeTicket`
2. 在订阅和发送登录邮件链路生成 bridge ticket
3. 验证链接回调时标记 bridge 为已验证
4. 订阅页前端轮询 bridge 状态
5. bridge 验证成功后显示一键登录按钮
6. 门户页加入持久登录恢复
7. 调整账号/退出入口文案和层级

结果：

1. 手机或电脑点邮件都可以完成验证
2. 另一台设备能自动感知并直接登录
3. 用户下次访问默认可自动进入门户

### 日志 2：推送节奏与订阅页风格升级

问题背景：

1. 旧推送模型虽然有分级，但摘要窗口偏长，不够贴近“近实时但不轰炸”的目标。
2. 订阅页过于像工具页，缺少产品层次感和价值说明。
3. 邮件内容缺少“为什么你会收到这封”的解释，用户心智不够稳。

设计决策：

1. 保留即时推送，但把普通新课放入更短周期的摘要窗口
2. 增加 `delivery_mode`，让邮件标题和通知事件可以表达实际推送模式
3. 订阅页重构为桌面双栏、移动单列的 Apple 风格落地页

实现步骤：

1. 在调度器中加入可配置摘要窗口
2. `_do_push()` 改为接受 `delivery_mode`
3. 邮件发送函数支持根据 `delivery_mode` 输出不同主题和导语
4. 邮件正文新增订阅偏好解释块
5. 订阅页首屏改版，加入说明区与更清晰的行动提示

结果：

1. 即时提醒与低打扰摘要已经开始分流
2. 订阅页第一屏已经能承载“产品说明 + 行为入口”
3. 后续可继续按同一视觉方向重构门户页

## 当前产品现状

### 已具备能力

1. 自动抓取博雅课程并写入数据库
2. 邮箱订阅、验证、退订
3. 无密码 magic link 登录
4. 长久登录与跨设备验证后登录

8. 管理台与 Nginx 访问控制分离

### 当前体验状态

1. 核心链路已经可用：
   - 填邮箱
   - 收到验证/登录邮件
   - 任意设备完成验证
   - 自动或一键进入门户
2. 登录体验已经从“每次重输邮箱”升级到“设备记住身份”
3. 推送体验已经从“单纯发信”升级到“即时提醒 + 摘要汇总”
4. 订阅页已经开始接近正式产品页，但门户页视觉层次仍有继续提升空间

### 当前仍存在的限制

1. 订阅页和部分旧模板文件存在编码历史包袱，后续继续改中文文案时仍需要小心处理
2. 门户页虽然完成了登录体验升级，但视觉上还没有完全和订阅页统一
3. 通知中心目前能看到通知记录，但还没有直观展示“这封通知属于哪种推送模式”
4. 推送聚合是系统级策略，还没有进一步做静默时段、暂停 24 小时等用户级控制

## 预期实现

预期中的完整产品形态应当是：

1. 用户不需要理解系统实现细节，只需要输入邮箱即可使用。
2. 验证、登录、长期保持登录都足够自然，不产生“我是不是又掉登录了”的疑问。
3. 推送既及时，又不过度轰炸邮箱。
4. 订阅页和门户页在视觉上统一，接近完整产品而不是拼装工具。
5. 用户能清楚理解：
   - 为什么收到这封邮件
   - 这封邮件紧不紧急
   - 我下一步该做什么
